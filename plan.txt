“Familiar” se usa en inglés para una criatura o compañero que acompaña y ayuda a una persona. Es más amplio que “pet”, por lo que permite perros, gatos, robots, fantasmas, slimes, avatares y personajes sin limitar el proyecto a animales.

La familia de nombres quedaría así:

Componente	Nombre
Aplicación	OpenFamiliar Desktop

Herramienta CLI	familiar
Formato de paquete	.familiar
Extensión de editor	OpenFamiliar Companion
Mascota inicial	Perrito Tech
Futuro catálogo	Familiar Gallery

El nombre no contiene “ChatGPT”, “Gemini”, “Grok”, “Codex” ni otra marca de proveedor, así que no nos ata a ninguna compañía ni da la impresión de ser un producto oficial.

Antes de hacerlo público se deberá realizar una revisión formal de:

Repositorios y organizaciones de GitHub.
npm, crates.io y otros registros.
Dominios.
Marcas registradas relevantes.

La revisión inicial del nombre no sustituye una búsqueda legal de marcas.

Decisión sobre la base del proyecto

Mi recomendación es:

Proyecto nuevo, no fork.

Podemos estudiar las ideas de OpenPets, CoPet, Clawd on Desk, OpenCode Companion y DeskPet, pero el núcleo de OpenFamiliar será escrito como una implementación propia.

Esto tiene varias ventajas:

Identidad y arquitectura propias.
Historial limpio.
Control completo de licencia.
Mejor valor como portafolio.
Posibilidad de aceptar contribuciones sin arrastrar restricciones inesperadas.
Libertad para separar mascotas, proveedores, workspace y agentes mediante SDKs.
Revisión de proyectos y licencias
OpenPets

OpenPets utiliza MIT. Podemos estudiar su arquitectura de plugins e incluso adaptar código concreto siempre que conservemos sus avisos de copyright y licencia.

CoPet

CoPet también utiliza MIT. Es una referencia especialmente útil para Tauri, paquetes de mascotas, hooks y compatibilidad con agentes como Codex, Antigravity, OpenCode y Gemini. Su repositorio declara Tauri, Rust y React, además de importación de paquetes personalizados.

DeskPet de Ice-teapop

Su código utiliza MIT, pero deja expresamente fuera determinados recursos provenientes de Clawd. Podemos estudiar el router de proveedores y las protecciones de seguridad, pero no copiar recursos excluidos.

Clawd on Desk

El código está bajo AGPL-3.0 y sus recursos gráficos tienen condiciones independientes: muchos están reservados y no pueden reutilizarse. No debemos copiar código, sprites, sonidos ni temas de este proyecto para un núcleo Apache-2.0. Lo utilizaremos solamente como referencia conceptual y para identificar comportamientos que podamos implementar de forma independiente.

OpenCode Companion

El repositorio revisado no contiene un archivo LICENSE en la ruta esperada y su README únicamente remite a “detalles de licencia”. Sin una licencia explícita, no debemos copiar su código ni sus recursos. Podemos estudiar públicamente el concepto, pero la implementación debe ser propia.

Open-LLM-VTuber

El núcleo revisado usa MIT, pero sus modelos Live2D tienen licencias independientes. Esto demuestra por qué OpenFamiliar debe separar claramente código, arte y paquetes de personajes.

Definición del producto

OpenFamiliar será una plataforma local para crear y ejecutar mascotas que puedan:

Vivir como ventana transparente sobre el escritorio.
Reaccionar a estados de agentes de código.
Recibir preguntas mediante texto.
Conectarse con modelos locales o APIs oficiales.
Obtener contexto de workspaces autorizados.
Mostrar exactamente qué información se enviará.
Solicitar aprobación antes de modificar archivos o ejecutar comandos.
Integrarse con editores compatibles mediante extensión o MCP.
Permitir crear nuevas mascotas sin modificar el núcleo.
Tres modos de seguridad
Chat
Puede:
- Conversar.
- Usar el modelo seleccionado.

No puede:
- Leer el workspace.
- Ejecutar comandos.
- Modificar archivos.
Read-only
Puede:
- Leer archivos autorizados.
- Consultar Git.
- Ver el archivo activo.
- Explicar código.

No puede:
- Escribir archivos.
- Ejecutar comandos mutables.
Agent
Puede:
- Proponer cambios.
- Mostrar diffs.
- Solicitar ejecución de comandos.
- Delegar tareas a un CLI.

Requiere:
- Aprobación explícita.
- Registro local de la acción.

MCP se utilizará como interfaz estandarizada para exponer herramientas y recursos de contexto, empezando por transporte local. El protocolo define precisamente herramientas y recursos para conectar aplicaciones de IA con sistemas externos.

Tauri con Rust y React ya ha demostrado ser viable para una mascota local ligera en CoPet, pero nuestra implementación será independiente.
